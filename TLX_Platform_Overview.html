Trade Link Exchange — แพลตฟอร์มกลางแลกเปลี่ยนเอกสารทางการค้า
กำกับโดย สพธอ. (ETDA) End-to-End Encryption (JWE / RFC 7516)

1บทนำ

TLX (Trade Link Exchange) คือแพลตฟอร์มกลางที่ใช้แลกเปลี่ยนเอกสารทางการค้าระหว่างผู้ประกอบการ ภายใต้การกำกับดูแลของ สพธอ. (ETDA)

หลักสำคัญ: ระบบเข้ารหัสแบบ End-to-End ด้วย JWE (RFC 7516) ทำให้ TLX ไม่สามารถอ่านเนื้อหาเอกสารได้ เก็บเฉพาะ Audit Evidence (Hash, Signature, Metadata) เพื่อความปลอดภัย เชื่อถือได้ และตรวจสอบย้อนหลังได้

6กระบวนการหลัก (Core Flows)

6.1 Participant Registration + KYC/KYB

ผู้ประกอบการสมัครผ่าน Portal → TLX Admin ตรวจสอบ → Approve/Reject → ระบบสร้าง Cognito User + Secret + Invitation Email

6.2 Key Generation + Registration

Generate RSA-4096 ที่ฝั่ง Client → ส่ง Public JWK ลงทะเบียน → Private Key ไม่เคยส่งให้ TLX

6.3 Key Rotation (Overlap Window)

Key Lifecycle: ACTIVE → DEPRECATED → REVOKED 1. Generate New Key Pair 2. PUT /keys/{old}/rotate (Atomic Transaction) 3. Overlap Window — old key ใช้ decrypt ได้ / new key ใช้ encrypt 4. GET /keys/{old}/drain-status (รอจน in-flight = 0) 5. POST /keys/{old}/complete-rotation → REVOKED

6.4 Partner Pairing

ค้นหา → ส่งคำขอ → Email แจ้ง → Verification รหัส 6 หลัก → Accept → พร้อมแลกเปลี่ยนเอกสาร

6.5 Message Exchange — Push Mode

  1. ดึง Public Key ของผู้รับ
  2. สร้าง Payload ตาม ETDA Format
  3. Canonicalize (RFC 8785) → JCS-SHA256
  4. Encrypt ด้วย Public Key ผู้รับ → JWE Token
  5. POST /v1/messages → Validate (Auth, Active, Matching, Schema, Idempotency)
  6. Audit Evidence (Hash, Key ID, Timestamp)
  7. Route → HTTP POST → Endpoint (Retry 3x → DLQ)

6.6 Message Exchange — Pull Mode

Receiver Poll Inbox → ดึงเอกสาร → Decrypt ด้วย Private Key → Verify Hash → DELETE (TTL 30 วัน)

7Transaction Flow — Full Purchase Lifecycle

  1. Product Catalog: Seller → PRODUCT_CATALOG → Buyer
  2. Quotation: QUOTATION ↔ QUOTATION_CONFIRM
  3. Order: PURCHASE_ORDER → SALES_ORDER
  4. Delivery: DELIVERY_ORDER → RECEIPT
  5. Invoice: Seller → INVOICE → Buyer
  6. Billing & Payment: INVOICE_COMPLETE → AR_BILL
  7. Adjustments: CREDIT_NOTE / DEBIT_NOTE (ถ้ามี)

8Message Validation Pipeline

Structural (JSON Schema)

  • Message ID (UUID), Sender/Receiver Tax ID (13 หลัก)
  • Document Type (1 ใน 12), Timestamp (ISO 8601), Schema Version

Semantic

  • Sender Active, Receiver มีอยู่ และรองรับ Document Type
  • Matching/Pairing มีอยู่, Message ID ไม่ซ้ำ (Idempotency)

Schema Versioning

  • เก็บใน Schema Registry (PostgreSQL)
  • Admin จัดการผ่าน Portal ไม่ต้องแก้ source code
  • รองรับหลาย version พร้อมกัน (backward compatible)

12Security Architecture

Authentication Pipeline (5 ชั้น)

  1. CloudFront + WAF (OWASP Top 10)
  2. Elastic Load Balancing (Health Check)
  3. Cognito JWT Validation
  4. Custom Authorizer (Participant Scope + ACTIVE only)
  5. TLS 1.2+ ทุก Connection

Key Management & Canonical Hashing

  • Private Key สร้างและเก็บฝั่ง Client เท่านั้น
  • Lifecycle: ACTIVE → DEPRECATED → REVOKED + Overlap Window
  • Canonicalize ด้วย JCS (RFC 8785) ก่อน Hash → SHA-256
  • Verify ย้อนหลัง: POST /v1/messages/{id}/verify-hash
