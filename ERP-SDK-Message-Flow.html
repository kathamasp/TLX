SDK Capabilities — ความสามารถหลักของ ERP SDK
SDK ที่ติดตั้งฝั่ง ERP ต้องรองรับฟังก์ชันทั้ง 7 ข้อต่อไปนี้ เพื่อให้สามารถแลกเปลี่ยนข้อมูลได้อย่างปลอดภัยและเป็นมาตรฐาน
01
Generate Message
สร้าง JSON message ตามรูปแบบมาตรฐานที่ สพธอ. กำหนด
02
Validate Message
ตรวจสอบโครงสร้างและความถูกต้องของ message ก่อนส่ง/หลังรับ
03
Generate Key Pair
สร้างคู่กุญแจ Public/Private key สำหรับ ERP แต่ละ Participant
04
Upload Public Key
ส่ง Public key ไปเก็บที่ TLX ผ่าน POST endpoint /keys/me
05
Fetch Partner Key
ดึง Public key ของคู่ค้าจาก TLX ในรูปแบบ JWK RFC 7517
06
Encrypt (JWE)
เข้ารหัส JSON message ด้วย Public key ของผู้รับ RFC 7516
07
Decrypt (JWE)
ถอดรหัส JWE ที่ได้รับด้วย Private key ของตัวเอง RFC 7516
🔑 Key Setup Flow — การตั้งค่ากุญแจเข้ารหัส (ทำครั้งเดียว)
ก่อนที่ ERP จะสามารถส่ง/รับข้อมูลแบบเข้ารหัสได้ ต้องผ่านขั้นตอนการตั้งค่ากุญแจก่อน
1
ERP เริ่มต้นสมัครใช้งานระบบ
ERP Participant ลงทะเบียนและได้รับ participant_id จาก TLX
2
SDK Generate RSA / EC Key Pair
SDK สร้างคู่กุญแจ privateKey + publicKey สำหรับ Participant นี้โดยเฉพาะ
• Algorithm แนะนำ: RSA-OAEP-256 หรือ ECDH-ES
• Private key เก็บไว้ใน ERP ภายใน (ไม่ส่งออกไปไหน)
3
SDK Export Public Key เป็นรูปแบบ JWK
แปลง Public key เป็น JSON Web Key (JWK) ตาม RFC 7517
ตัวอย่าง: { "kty":"RSA", "use":"enc", "alg":"RSA-OAEP-256", "n":"...", "e":"AQAB" }
4
TLX Upload Public Key → POST /keys/me
SDK ส่ง JWK ไปที่ TLX endpoint: POST /api/v1/keys/me
TLX บันทึก Public key เชื่อมกับ participant_id ของ ERP นั้น
5
TLX TLX ยืนยันและเก็บ Public Key
TLX ตอบกลับ 200 OK พร้อม key_id สำหรับอ้างอิงในอนาคต
✅ ERP Participant พร้อมส่ง/รับข้อมูลเข้ารหัสแล้ว
📤 Send Message Flow — การส่งข้อมูล (Sender Side)
Flow นี้ทำงานทุกครั้งที่ ERP ต้องการส่งข้อมูลไปยังคู่ค้า โดยข้อมูลจะถูกเข้ารหัสด้วย Public key ของผู้รับ
1
ERP สร้างข้อมูลธุรกิจ (Business Data)
ERP เตรียมข้อมูล เช่น Invoice, PO, GR ที่ต้องการส่งให้คู่ค้า
2
SDK Generate JSON Message (สพธอ. Standard)
SDK แปลงข้อมูลเป็น JSON ตามโครงสร้างมาตรฐาน สพธอ.
ระบุ Header: sender_id, receiver_id, message_type, timestamp
3
SDK Validate Message Structure
ตรวจสอบว่า JSON ถูกต้องตาม Schema ที่กำหนด
• ตรวจ required fields, data types, format
• หาก Validate ไม่ผ่าน → ส่ง error กลับ ERP และหยุดการทำงาน
4
TLX Fetch Partner's Public Key → GET /keys/{partner_id}
SDK ดึง Public key ของ Receiver จาก TLX: GET /api/v1/keys/{receiver_id}
TLX ส่งคืน JWK ของ Receiver RFC 7517
💡 Cache ได้ชั่วคราวเพื่อลด API calls
5
SDK Encrypt Message → JWE RFC 7516
เข้ารหัส JSON payload ด้วย Receiver's Public Key
• Key Encryption: RSA-OAEP-256 หรือ ECDH-ES
• Content Encryption: A256GCM
ได้ผลลัพธ์เป็น Compact JWE: header.key.iv.ciphertext.tag
6
NETWORK ส่ง Encrypted JWE ผ่าน TLX
ERP SDK ส่ง JWE payload ไปยัง TLX Message endpoint
TLX ทำหน้าที่เป็น Message Broker → ส่งต่อให้ Receiver
7
TLX TLX บันทึก Log และส่งต่อ
TLX บันทึก audit log, ตรวจสอบสิทธิ์ match คู่ค้า แล้วส่ง JWE ต่อให้ Receiver
✅ Sender ได้รับ acknowledgement กลับมา
📥 Receive Message Flow — การรับข้อมูล (Receiver Side)
Flow นี้ทำงานเมื่อ ERP ได้รับ Encrypted JWE message จาก TLX — ถอดรหัสด้วย Private key ของตนเอง
1
TLX TLX ส่ง Encrypted JWE มาให้ Receiver
TLX ส่ง JWE payload มาให้ ERP Receiver ผ่าน Webhook / Polling endpoint
ข้อมูลที่ได้รับเป็น Compact JWE: header.key.iv.ciphertext.tag
2
SDK Verify JWE Header
อ่าน JWE Header เพื่อตรวจสอบ alg และ enc ที่ใช้
ตรวจสอบว่า kid ตรงกับ Key ที่ตัวเองถือ
3
SDK Decrypt JWE → JSON Plaintext RFC 7516
ถอดรหัส JWE ด้วย privateKey ของตัวเอง
• ถอดรหัส Content Encryption Key (CEK) ก่อน
• จากนั้น decrypt ciphertext ด้วย CEK + IV
ได้ผลลัพธ์เป็น JSON plaintext
4
SDK Validate Decrypted JSON Message
ตรวจสอบโครงสร้าง JSON ที่ถอดรหัสแล้วว่าถูกต้องตาม Schema สพธอ.
• ตรวจ message integrity
• ตรวจสอบ sender_id ตรงกับคู่ค้าที่ match ไว้หรือไม่
• ตรวจ timestamp ป้องกัน Replay Attack
5
ERP นำข้อมูลเข้าสู่ระบบ ERP
SDK ส่ง JSON plaintext ที่ผ่าน Validate แล้วให้ ERP ประมวลผล
ERP บันทึก Invoice / PO / GR เข้าระบบตามปกติ
✅ ส่ง acknowledgement กลับไปยัง TLX
🌐 TLX New Endpoints — 2 เส้นที่ต้องเพิ่ม
เพื่อรองรับระบบ Key Management TLX ต้องเพิ่ม endpoint ใหม่ 2 เส้น ดังนี้

🔑 Key Management Endpoints

POST
/api/v1/keys/me

Upload / Update Public Key — ERP ส่ง JWK (Public key) ของตัวเองมาเก็บที่ TLX
Request body: { "jwk": { ... } }  |  Response: { "key_id": "...", "created_at": "..." }
Auth: Bearer token ของ Participant นั้น

GET
/api/v1/keys/{participant_id}

Fetch Partner's Public Key — ดึง JWK ของ Participant ที่ match แล้วเท่านั้น
Response: { "participant_id": "...", "jwk": { "kty":"RSA", "use":"enc", "alg":"RSA-OAEP-256", ... } }
Format: JWK ตาม RFC 7517  |  Auth: เฉพาะ Participant ที่ match กันเท่านั้น

🔒 Security Considerations สำหรับ TLX

• Public key จะถูก associate กับ participant_id ที่ authenticated แล้วเท่านั้น
• GET /keys/{participant_id} ต้องตรวจสอบว่า requester กับ participant_id นั้น match กันใน TLX แล้ว
• ควรมี Key rotation support — เมื่อ POST ใหม่ ให้ invalidate key เก่าหรือ version ไว้
• Log ทุกครั้งที่มีการ upload/fetch key เพื่อ audit trail
• Rate limit endpoint เพื่อป้องกัน enumeration attack

📐 JWE Structure Summary

Compact Serialization (RFC 7516):
BASE64URL(JWE Header) . BASE64URL(Encrypted Key) . BASE64URL(IV) . BASE64URL(Ciphertext) . BASE64URL(Auth Tag)

แนะนำ Algorithm:
• alg: RSA-OAEP-256 (Key Encryption)
• enc: A256GCM (Content Encryption)
